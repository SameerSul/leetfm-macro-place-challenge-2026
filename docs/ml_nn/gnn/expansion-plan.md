# GNN Expansion Plan

The first production-safe role for the GNN is candidate ranking. That is still
the required starting point, but the subsystem should expand into a hierarchy
flow controller once the ranker is validated.

The core rule does not change: learned outputs may propose, rank, skip, or
budget work, but deterministic legality, fixed-macro, bounds, hierarchy-region,
hierarchy-quality, and exact-proxy gates remain authoritative.

## Expanded Roles

### 1. Candidate Ranking

Initial role. The GNN scores candidates already generated by existing hierarchy
operators.

Applies to:

- hard relocation;
- soft relocation;
- hard/hard swaps;
- hard/soft swaps;
- soft/soft swaps;
- cluster decompression;
- coldspot tightening.

Gate:

- The GNN only changes ordering.
- Exact scoring and existing accept gates still decide moves.

### 2. Candidate Pool Expansion

The GNN may propose extra candidates in addition to heuristic candidates.

Examples:

- add swap targets that are graph-near but spatially non-obvious;
- add relocation targets near connected clusters;
- add decompression directions suggested by cluster/net pressure;
- add coldspot targets that preserve connected subsystem shape.

Gate:

- Proposed candidates are unioned with heuristic candidates.
- Existing legality and exact-proxy checks remain mandatory.
- `HIER_GNN_TOP_K` and per-operator caps control runtime.

### 3. Operator Selection

The GNN predicts which hierarchy operator is worth running next for a given
state.

Possible actions:

- run relocation;
- run swaps;
- run decompression;
- run coldspot tightening;
- run micro-shift;
- skip an operator.

Gate:

- Operator selection starts as advisory logging.
- Default-off execution can skip only after benchmark validation shows runtime
  or quality improvement.

### 4. Region Guidance

The GNN predicts hierarchy-region adjustments.

Possible outputs:

- expand a cluster region;
- shrink a region;
- shift a region toward lower congestion;
- change region escape bias;
- adjust bridge-soft corridor width.

Gate:

- Region changes must still respect bounds and fixed macros.
- Region changes must be exact-proxy and hierarchy-quality gated downstream.

### 5. Soft Macro Role Guidance

The GNN predicts soft macro ownership and bridge roles.

Possible outputs:

- owned cluster id;
- bridge cluster set;
- soft corridor preference;
- whether a soft macro should co-move with decompression or coldspot proposals.

Gate:

- Start as a candidate-role proposal compared against deterministic roles.
- Do not replace deterministic soft-role derivation until closed-loop evidence
  shows improvement.

### 6. Acceptance Risk Surrogate

The GNN predicts whether a candidate is likely to fail before expensive exact
scoring.

Possible heads:

- exact-proxy-fail probability;
- hierarchy-quality-fail probability;
- region-escape risk;
- legality-risk signal;
- downstream congestion-risk signal.

Gate:

- Risk predictions can prioritize or deprioritize scoring.
- They must not be the final accept/reject decision.

### 7. Budget Allocation

The GNN predicts how much work to spend per operator.

Possible outputs:

- relocation candidate width;
- swap `K`;
- decompression rounds;
- coldspot rounds;
- whether to run density and congestion variants;
- per-pass time budget.

Gate:

- Budget predictions start as logging.
- Default-off integration must preserve the one-hour full-suite budget with
  margin.

### 8. Diagnostics And Benchmark Triage

The GNN can produce embeddings and risk reports for analysis even before online
use.

Useful outputs:

- clusters likely to regress;
- soft-heavy congestion failures;
- benchmarks where hierarchy and proxy conflict;
- operator/pass value estimates.

Gate:

- Diagnostics are always default-off and non-mutating.

## Implementation Order

Recommended sequence:

1. G3 baseline candidate ranker.
2. G4 hetero macro-net candidate ranker.
3. Auxiliary heads for proxy delta, accept probability, hierarchy-quality delta,
   and rejection reason.
4. Candidate pool expansion.
5. Operator selection in advisory/logging mode.
6. Region guidance and soft-role guidance.
7. Budget allocation.
8. Closed-loop policy over hierarchy states, still using deterministic gates.

## Additional Trace Requirements

To support expanded roles, add traces for:

- operator state before and after each pass;
- operators skipped by heuristic rules;
- per-pass runtime and candidate count;
- candidate pool sizes before/after GNN expansion;
- region boxes before/after guidance;
- deterministic soft-role labels and any GNN-proposed roles;
- exact score count saved or added by risk/budget decisions.

## Production Boundary

The expanded GNN is a hierarchy-flow assistant, not an external placer. It may:

- rank;
- propose;
- select;
- budget;
- diagnose.

It may not:

- bypass legality;
- move fixed macros;
- accept moves without exact gates;
- emit final placements directly;
- replace grouped DREAMPlace or hierarchy legalization.
